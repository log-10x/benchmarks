The pattern ID you can plot:
log10x vs Drain3

Both tools collapse repetitive log lines into templates. Only log10x gives each line a pattern ID that stays identical across files, hosts, and order — the thing that lets you ask "when did this event start?" and plot one event over time. And its encoding is reversible: Drain3 mines templates and discards the values, while log10x keeps them and decodes the exact original back. Identical raw input to both, no per-source configuration; every number below is reproducible from the commands at the bottom.

01 Pattern stability

The differentiator: only a stable ID lets you ask "when did this event start?" or plot one event over time. Split each dataset into halves A and B; run each tool on A, B, A+B, B+A, and check: does a given line get the same pattern ID regardless of what else is in the file? Below: % of lines whose pattern is unchanged when its half is processed alongside other content (context-invariance, worst of the two directions).

log10x's template hash is a pure function of the line's structure — same line, same hash, any file, any order (100% on all 16). Drain3 is an online learner: its templates generalize as it consumes the stream, so identity depends on history.

02 Reversibility

A stable ID names the event; reversibility means the exact events are still there behind it. Drain3 mines templates and discards the values — out of the box it reconstructs nothing. log10x keeps every value, typed, beside the hash and decodes the exact original back. Bars show byte-exact reconstruction per dataset (log10x ran a true round trip: kept only templates.json + encoded.log, deleted the rest, decoded back).

log10x = 100% (full track) Drain3 byte-exact % sorted by Drain3 · hover for detail

Drain3's shortfall is only whitespace: a token-aligned rebuild recovers 100% of the content on all 16, but its tokenizer collapses space runs (Jul  1 → Jul 1), so padded lines are not byte-exact. A footnote next to the structural point: one tool stored the events, the other kept only the shapes.

03 Typed extraction

The reason the round trip works: every variable is pulled into its own typed field. One HDFS line and its encoded.log record:

raw line081109 203615 148 INFO dfs.DataNode$PacketResponder: PacketResponder 1 for block blk_38865049064139660 terminating
log10x encoded record~9PzqOgnWcl,081109,203615,148,DataNode,PacketResponder,1,38865049064139660
~9PzqOgnWcl template hash 081109 date 203615 time 148 pid DataNode component PacketResponder subclass 1 responder 38865049064139660 block id

Drain3's equivalent is the template string PacketResponder <*> for block blk_<*> terminating with the values dropped. The hash is content-addressed; the fields are typed and queryable.

05 How to reproduce

Data

git clone --depth 1 https://github.com/logpai/loghub.git

16 *_2k.log datasets, ~2,000 raw lines each. Normalize line endings to LF, then feed the whole raw line to both tools — no per-source header regex.

Drain3 0.9.11

pip install drain3==0.9.11; default config. Mine templates, then reconstruct by token alignment: Drain clusters are length-homogeneous, so each variable is recoverable by position (bolted on, since Drain3 doesn't reconstruct).

log10x engine 1.1.4

docker run … log10x/pipeline-10x:latest @tenx-encode.config.yaml reads a log file and writes templates.json + encoded.log; a second decode pass turns those back into decoded.log. No API key for local file I/O.

Metrics

Lossless % = byte-exact reconstructed lines ÷ total. Templates = distinct patterns. Stability = % of lines keeping the same pattern across the A / B / A+B / B+A split.
