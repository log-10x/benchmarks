A log parser that keeps every byte:
log10x vs Drain3

Both tools collapse repetitive log lines into templates. Drain3 is a template miner — it finds the shapes and discards the lines. log10x is a lossless codec — same template view, but it reconstructs the exact original stream. Identical raw input to both, no per-source configuration. Every number below is reproducible from the commands at the bottom.

01 Losslessness

Reduce the file, then rebuild it: what fraction of lines come back byte-for-byte identical? For log10x we also did a true round trip — kept only templates.json + encoded.log, deleted the rest, and decoded that file back to the exact original.

Drain3's loss tracks whitespace: it splits on spaces and collapses runs of them, so Jul  1 → Jul 1. Every Mac/Windows line has a whitespace run → 0% recoverable.

02 Pattern stability

Split each dataset into halves A and B; run each tool on A, B, A+B, B+A. Does a given line get the same pattern ID regardless of what else is in the file? Below: % of lines whose pattern is unchanged when its half is processed alongside other content (context-invariance, worst of the two directions).

log10x = 100% (content-addressed hash) Drain3 context-invariant % hover for detail

log10x's template hash is a pure function of the line's structure — same line, same hash, any file, any order (100% on all 16). Drain3 is an online learner: its templates generalize as it consumes the stream, so identity depends on history.

03 Typed extraction

The reason the round trip works: every variable is pulled into its own typed field. One HDFS line and its encoded.log record:

raw line081109 203615 148 INFO dfs.DataNode$PacketResponder: PacketResponder 1 for block blk_38865049064139660 terminating
log10x encoded record~9PzqOgnWcl,081109,203615,148,DataNode,PacketResponder,1,38865049064139660
~9PzqOgnWcl template hash 081109 date 203615 time 148 pid DataNode component PacketResponder subclass 1 responder 38865049064139660 block id

Drain3's equivalent is the template string PacketResponder <*> for block blk_<*> terminating with the values dropped. The hash is content-addressed; the fields are typed and queryable.

05 How to reproduce

Data

git clone --depth 1 https://github.com/logpai/loghub.git

16 *_2k.log datasets, ~2,000 raw lines each. Normalize line endings to LF, then feed the whole raw line to both tools — no per-source header regex.

Drain3 0.9.11

pip install drain3==0.9.11; default config. Mine templates, then (bolted on, since Drain3 doesn't) a per-line parameter store via extract_parameters to attempt reconstruction.

log10x engine 1.1.4

docker run … log10x/pipeline-10x:latest @tenx.config.yaml. Reads a log file, writes templates.json, encoded.log, reconstructed.log. No API key for local file I/O.

Metrics

Lossless % = byte-exact reconstructed lines ÷ total. Templates = distinct patterns. Stability = % of lines keeping the same pattern across the A / B / A+B / B+A split.
